🏗️ Generators vs Harness Architecture

Understanding the separation of concerns and exploring plugin architecture

🔧 What the Harness Can Do

💡 Key Insight

The harness is essentially a code-writing agent. It can read existing code, understand patterns, and write new code. But it needs guidance in the form of:

  • Clear task descriptions
  • Examples of similar implementations
  • Specific file locations
  • Expected interfaces/types

🎨 What ARE Generators?

The Generator Philosophy

Generators are "dumb" code-producing functions. They don't think, don't learn, don't improve themselves. They simply:

  1. Take a prompt
  2. Call LLM with system prompt
  3. Return generated code

Current Generator Structure

├── p5/
│   ├── P5Generator.ts          # Main generator

Each generator:

- Has a generate(prompt) method
- Uses domain-specific system prompts
- Returns { code, explanation }

🔌 Generators as Skills/Plugins

💡 THE IDEA: Plugin Architecture

Instead of hardcoded generators, make them discoverable plugins:

  • Add new domains without touching core code
  • Community can create generators
  • Load only what you use
  • Versioned independently

Proposed Plugin Structure

plugins/
├── core/                           # Built-in plugins
│   ├── p5/
│   │   ├── manifest.json          # Plugin metadata
│   │   ├── generator.ts           # Generator implementation
│   │   ├── prompts/               # System prompts
│   │   ├── templates/             # Code templates
│   │   └── validators/            # Domain-specific validation
│   ├── three/
│   ├── glsl/
│   └── ...
│
└── community/                      # Third-party plugins
    ├── unity/
    │   ├── manifest.json
    │   ├── generator.ts
    │   └── README.md
    ├── unreal/

    └── ...

Plugin Discovery & Loading

// PluginManager.ts
class PluginManager {
  private plugins: Map = new Map();

  async loadPlugin(path: string): Promise {
    const manifest = await this.readManifest(path);
    const plugin = await import(manifest.entry);
    
    // Validate plugin interface
    if (!this.validatePlugin(plugin)) {
      throw new Error(`Invalid plugin: ${manifest.id}`);
    }
    
    this.plugins.set(manifest.id, plugin);
  }

  getGenerator(domain: string): GeneratorPlugin | undefined {
    // Find plugin by supported domain
    for (const [id, plugin] of this.plugins) {
      if (plugin.manifest.supportedDomains.includes(domain)) {
        return plugin;
      }
    }
  }

  listPlugins(): PluginInfo[] {
    return Array.from(this.plugins.values()).map(p => p.manifest);
  }
}

⚖️ Trade-offs: Built-in vs Plugin

Aspect Current (Built-in) Proposed (Plugin)
Installation All included, bloated Install only what you use
New Domain Edit core code Drop in plugin folder
Community PR to core repo Publish plugin package
Versioning Tied to Liminal version Independent versions
Reliability Tested with core Variable quality
Complexity Simple More complex (loader, validation)
Startup Time Slower (loads all) Faster (lazy loading)

🎯 Recommendation

Hybrid Approach

Phase 1 (Now): Keep core generators built-in for stability

Phase 2 (Future): Add plugin system for extensibility

The harness can help build the plugin system by:

  1. Creating the PluginManager class
  2. Converting existing generators to plugin format
  3. Creating plugin templates for new domains
