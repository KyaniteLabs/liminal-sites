# Manual QA and Marketing Recording Guide

This guide exists to keep human time focused on what only humans can judge: taste, motion quality, audio feel, recognizability, and recording usefulness.

## Machine prep before humans start

Run these first:

```bash
pnpm build
pnpm proof:live-provider-smoke -- --timeout-ms=120000
pnpm proof:live-model-assimilation
pnpm proof:live-creative-domains -- --all --timeout-ms=180000
node bin/liminal release gate --output .omx/final-release-gate-all-domains.json
pnpm qa:creative-domains -- --input .omx/proof/domain-gauntlet-live.json --open
```

Humans should not begin until the release gate says `Ready: YES` and the QA cockpit opens.

## What the cockpit gives testers

- One browser page with every generated domain artifact.
- One-click machine checks for load errors, iframe inspection, console errors, and visible element counts.
- Per-domain expectations so testers do not invent criteria mid-session.
- Checkboxes for the human-only pass.
- Notes fields and a copyable bug report so failures are not retyped from scratch.
- A recording order in `checklist.md` so marketing capture is predictable.

## What humans should spend time on

Only these questions:

1. Does it look/sound/feel good enough to show?
2. Is the output visibly/audibly the requested domain?
3. Is there obvious brokenness a machine missed?
4. Does this artifact help or hurt the first marketing recording?

If the answer is no, copy the cockpit report and attach a screenshot or screen recording.

## What humans should not spend time on

- Finding artifact paths.
- Re-running proof commands without a clear failure.
- Manually collecting provider/model provenance.
- Typing reproduction templates.
- Guessing which domains are in scope.

## Recording fallback

If live generation is slow during recording, use the latest `.omx/proof/live-creative-domains/*` artifacts and the cockpit. Say plainly that the artifacts were generated by the live all-domain proof run and show the receipt.
