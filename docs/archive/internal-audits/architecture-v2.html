Liminal Architecture V2

Corrected: Multi-Type Guardrails & Harness-Generator Relationship
🔧 Architecture Correction

The previous architecture incorrectly showed generators as "dumb" and implied they weren't improved by the harness. This was wrong.

Correct Understanding:

Generators don't have their OWN self-improving loop, but they ARE improved by the harness like everything else. The harness can fix generator code, prompts, validation - anything.

Guardrail Types:

There are multiple types of guardrails, not one. Code quality guardrails apply to everyone, but each generator needs its own OUTPUT aesthetic quality guardrails.

🛡️ Two Types of Guardrails
🔴 Type 1: Universal Code Quality Guardrails
APPLIES TO EVERYONE

Structural validation that all code must pass regardless of generator.

  • Syntax validation (balanced braces, parens)
  • Domain detection (p5, shader, three, etc.)
  • Size limits (min/max code length)
  • Forbidden patterns (eval, new Function)
  • Required functions (setup, draw for p5)
  • Basic structure completeness
Implementation:
src/core/CodeValidator.ts - Single validator used by all
🟣 Type 2: Per-Generator Aesthetic Guardrails
ONE PER GENERATOR

Output quality evaluation specific to the generator's domain.

  • P5: Visual composition, color harmony, animation
  • Tone: Audio synthesis quality, frequency balance
  • Shader: Visual complexity, raymarching quality
  • Three: 3D scene composition, lighting
  • Strudel: Rhythm patterns, musical structure
  • Hydra: Video synthesis chains, modulation
Implementation:
Each generator OR CreativeEvaluator needs domain-specific aesthetic checks

The harness doesn't just fix "the system around generators" - it can directly fix generator prompts, validation, and quality checks. M1 is a perfect example: it fixes Tone.js validation IN the CodeValidator that affects ToneGenerator outputs.
